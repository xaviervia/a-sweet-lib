What is this?

A Sweet Lib gives you the building blocks for a functional programming approach in JavaScript.
  • Only higher order functions
  • All functions with several arguments are partially applied
  • Only essential functions, the building blocks. More domain specific ones you can derive yourself
  • Easy to participate. All the functions are gists put as submodules in the lib repo, which means that you can comment on each function individually but also play around with alternative implementation in your own gists and submit them in the repo issues.

Why building this?

There is still space in the ecosystem for a library that takes the middle way between the hard core ones that offer to completely change your programming style with a hard core functional approach (such as Ramda) and the ones that just provide solutions to common problems without actually offering any philosophy accompanying them (such as Lodash).

A Sweet Lib aims to hit the sweet spot in the middle: offer a collection of well structured functions based on an idea, but also demystifying them. There is nothing odd about map or reduce and that should be obvious. Your library should tell you that. That's what A Sweet Lib does. Furthermore, A Sweet Lib doesn't need to sit as a global, hard to get rid of dependency in your projects: you can just copy paste the snippets. That's the main purpose of why it was built.

Not only the code is copy pasteable: the example tests are copy pasteable because they are generic as well. Since they are just functions using assert, they work in all the test libraries I am aware of. Only the path to the original function needs to be rewritten

Why are some functions only given as examples and not exposed?

It so happens that the functions in the examples are combination of building blocks functions. For example, a very common variation is to fix the arity of the target function while doing a compose, a flip, curry, etc. Many of these combinations lead to, well, combinatorics, which means that a comprehensive list of the derived functions will approximate factorial of the amount of building block functions. To support that is of course nonesense.

The given examples are carefully chosen to demonstrate common combinations, to provide semantic solutions to common problems (such as the all functions, which is just reduce(and)) and to excite your immagination about the posibilities.

If you think there is an example that matches the described criteria and it's missing from the lib, please write a gist and comment an issue with it.

Not an utilities library
That's why all functions are higher order functions. They are meant to help you build the logic of your program, not to build it for you.
